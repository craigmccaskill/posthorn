This page is the **canonical public roadmap**. The narrative lives here; the execution surface (open issues, in-flight work) lives in [GitHub Milestones](https://github.com/craigmccaskill/posthorn/milestones).

Posthorn ships in deliberately small, scoped releases. Every version has a locked spec before code is written. The "Worked" criterion for the project is two binary signals: at least one external user in production within 90 days of v1.0, and the Ghost admin-login round-trip via v1.2 SMTP ingress within 6 months.

## v1.0 — first public release

**Status:** released or imminent (see the [GitHub Releases page](https://github.com/craigmccaskill/posthorn/releases) for the live date).

**Scope:** HTTP form ingress, Postmark transport, full spam-protection stack, rate limiting, templating, JSON + redirect responses, standalone Docker container, optional Caddy adapter.

What you can do today:

- Drop in a self-hosted contact form on any host that blocks outbound SMTP
- Run Posthorn as a Docker sidecar next to your blog / static site / app stack
- Or load it as a Caddy v2 module if you're already running Caddy

[Get started →](/getting-started/quick-start/)

## v1.1 — API mode

[Track on GitHub →](https://github.com/craigmccaskill/posthorn/milestone/1)

The version where Posthorn becomes usable as **an internal mail API**, not just a contact-form gateway. Server-to-server callers (a worker, a backend job, a paid-event handler) can now hit Posthorn without needing browser-shaped defenses (Origin/Referer/honeypot).

- **API-key auth per endpoint** — `auth = "api-key"` mode, mutually exclusive with the form-mode defenses. Callers send `Authorization: Bearer <key>`. Rate limit still applies.
- **JSON content type on API-mode endpoints** — natural shape for server-to-server callers. Form-mode endpoints stay form-encoded only.
- **Idempotency keys** — standard `Idempotency-Key` header, 24-hour TTL, in-memory cache. Safe retries for payment / license-delivery / one-shot transactional flows. *Durable across restarts in v2.*
- **Batch send** — per-recipient template substitution against Postmark's `/email/batch` endpoint, hard cap aligned to Postmark's 500/batch limit, per-recipient response status.

No breaking config changes — every v1.0 form-mode config keeps working untouched. API-mode is opt-in per endpoint.

## v1.2 — multi-transport + operational maturity

[Track on GitHub →](https://github.com/craigmccaskill/posthorn/milestone/4)

The version that breaks Postmark-lock and makes Posthorn production-ready. Adding 4 transports is the moment operators care about metrics, health checks, dry-run validation, and tighter spam protection.

**Transports** (any of these can be configured per endpoint, alongside Postmark):

- **Resend** — alternative HTTP API provider
- **Mailgun** — alternative HTTP API provider
- **AWS SES** — AWS-native deployments
- **Outbound SMTP relay** — the historical baseline, for operators on hosts that don't block port 587 (Hetzner, OVH, etc.)

**Spam protection** (form-mode hardening on top of v1.0):

- CSRF tokens + time-based form tokens — catches automated abuse the v1.0 honeypot + Origin/Referer doesn't

**Observability + ergonomics:**

- `/healthz` endpoint for Docker / Kubernetes liveness probes
- `/metrics` endpoint (Prometheus exposition format) — submission count, latency histograms, error class breakdown
- Dry run mode — parse + validate without sending, useful in CI for customer-facing form template changes
- Named presets for `trusted_proxies` (`cloudflare`, etc.) on top of the v1.0 CIDR-list syntax
- IP-stripping option for GDPR contexts

## v1.3 — SMTP ingress

[Track on GitHub →](https://github.com/craigmccaskill/posthorn/milestone/2)

The strategic feature that completes the gateway thesis. **A TCP listener accepting SMTP from clients on the local network**, forwarding via whatever HTTP API transport is configured.

The use case: self-hosted apps that emit SMTP for transactional mail — Ghost, Gitea, Mastodon, Discourse, Matrix, NextCloud, Authentik — currently broken on cloud hosts that block outbound SMTP. v1.3 makes them work.

The architectural commitments are already locked in v1.0:

- The `Message` struct doesn't change — it's the boundary between ingress (form parser, SMTP MIME parser) and egress (transports)
- New `Ingress` interface — HTTP form ingress is the first instance (in v1.0); SMTP listener is the second
- New top-level config section `smtp_listener` alongside `endpoints`
- Caddy adapter does **not** get SMTP ingress — the standalone binary is the natural sidecar
- New threat model (open relay prevention, RCPT validation, sender allowlist, recipient caps, optional client-cert auth) gets its own spec rewrite in v1.3

## v2 — platform maturity

[Track on GitHub →](https://github.com/craigmccaskill/posthorn/milestone/3)

The architectural shift that unlocks operating Posthorn as a real mail platform: durable state, lifecycle awareness, deliverability hygiene.

**Storage layer:**

- **SQLite submission log** — every submission persisted, queryable
- **Retry queue across restarts** — failed sends survive a container restart and retry later

**Mail-platform features built on storage:**

- **Suppression list** — auto on hard bounces and spam complaints; admin API for inspection and manual entries. Refuses to send to suppressed addresses (returns 200 with `{"suppressed": true}` so callers treat it as final, not transient).
- **Durable idempotency** — the v1.1 in-memory cache becomes restart-survivable
- **Postmark lifecycle event callbacks** — delivery / bounce / open / click events forwarded to the originating caller with HMAC-signed webhook
- **Automatic unsubscribe link injection** — per-recipient signed tokens, hosted unsubscribe endpoint, RFC 8058 one-click headers (which Gmail and Yahoo now require for senders &gt;5000/day)
- **HTML body support** (alongside the originally-planned markdown body)
- **File attachments** — multipart form uploads forwarded to the transport
- **Multiple outputs per endpoint** — email + webhook + log fan-out for the same submission

The `Transport.Send` interface stays the synchronous primitive; the queue wraps it. v1.x deployments keep working unchanged.

## v3 — speculative

No date. Depends entirely on community traction:

- **Admin UI** — embedded web app for browsing submissions, viewing logs, retrying failed sends (requires v2's SQLite storage)
- **Proof-of-work spam challenge** — computational cost imposed on submitters; defeats botnet spam from many low-rate IPs that the v1.0 rate limiter can't catch
- **PGP encryption** — encrypt outbound mail to recipient public keys at the gateway

## How to influence the order

<Aside type="tip">
The roadmap order isn't set in stone after v1.0 ships. Adoption signals reshuffle priorities — if every early user asks for the same thing, that thing moves up.
</Aside>

If you want to push something forward:

1. **Open a [feature request](https://github.com/craigmccaskill/posthorn/issues/new?template=feature.yml)** with a concrete use case. Pitches like "wouldn't it be cool if..." get less weight than "I want to use Posthorn for X, but Y is missing."
2. **Comment on the relevant milestone or existing issue** with your use case. Adoption signal beats pure upvotes.
3. **Start a [discussion](https://github.com/craigmccaskill/posthorn/discussions)** for half-formed ideas where the right framing isn't clear yet.

## What's deliberately out of scope

These come up in conversation and are worth naming explicitly:

| Feature | Why not |
|---|---|
| Acting as a "real" MTA (MX records, inbound mail) | Posthorn is an authenticated relay, not a mail server. The v1.3 SMTP ingress accepts mail from known internal clients only. |
| Markdown body rendering | Templates are plain text. v2 may add markdown with explicit escaping. |
| Multi-tenancy / per-tenant config isolation | Post-v1 if at all — the project's audience is single-tenant self-hosters. |
| GUI installer / first-run wizard | The TOML / Caddyfile config is the source of truth; no UI in front of it. |
| Replacing Postmark / providers entirely (e.g., running our own SMTP outbound) | That's "build a postfix replacement" territory; out of scope forever. |
