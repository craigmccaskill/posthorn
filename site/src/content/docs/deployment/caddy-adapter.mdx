If you're already running Caddy as your front door, the Caddy adapter lets you skip the separate Posthorn process entirely. It's the same core gateway code, loaded as a Caddy v2 module (`http.handlers.posthorn`) and configured through Caddyfile directives.

<Aside type="note">
The Caddy adapter is fully supported and shares 100% of its pipeline with the standalone. But the standalone is the **primary** deployment shape. If you don't already run Caddy, [use Docker](/deployment/docker/) instead — it keeps the email-gateway concern decoupled from your reverse-proxy concern.
</Aside>

## Build a Caddy with Posthorn included

The adapter is published as a separate Go module: `github.com/craigmccaskill/posthorn/caddy`. Build a custom Caddy with it via `xcaddy`:

```bash
xcaddy build --with github.com/craigmccaskill/posthorn/caddy
```

That produces a `caddy` binary in the current directory. Verify the module loaded:

```bash
./caddy list-modules | grep posthorn
# http.handlers.posthorn
```

Requires Caddy 2.9+.

## Caddyfile syntax

The directive is `posthorn <path> { ... }`. It's a request handler — Caddy will route matching requests to Posthorn, which then runs the same pipeline as the standalone (body cap → method → content-type → origin → rate limit → parse → honeypot → required → email → template → send).

```nginx
example.com {
    posthorn /api/contact {
        to you@example.com
        from "Contact Form <noreply@example.com>"
        required name email message
        honeypot _gotcha
        allowed_origins https://example.com
        subject "Contact from {{.name}}"
        body "From: {{.name}} <{{.email}}>\n\n{{.message}}"
        rate_limit 5 1m
        redirect_success /thank-you

        transport postmark {
            api_key {env.POSTMARK_API_KEY}
        }
    }
}
```

### Directive grammar

```text
posthorn <path> {
    to              <email> [<email> ...]
    from            <name-and-email>
    reply_to_email_field <field>
    required        <field> [<field> ...]
    email_field     <field>
    honeypot        <field>
    allowed_origins <url> [<url> ...]
    max_body_size   <size>
    trusted_proxies <cidr> [<cidr> ...]
    subject         <template>
    body            <template-or-path>
    redirect_success <url>
    redirect_error   <url>
    log_failed_submissions {true|false}

    rate_limit      <count> <interval>

    transport <type> {
        # transport-specific settings
        api_key <value>
    }
}
```

### Mapping from TOML

| TOML | Caddyfile |
|---|---|
| `path = "/api/contact"` | `posthorn /api/contact` (the directive's first arg) |
| `to = ["a@b", "c@d"]` | `to a@b c@d` |
| `required = ["name", "email"]` | `required name email` |
| `[endpoints.rate_limit] count = 5; interval = "1m"` | `rate_limit 5 1m` |
| `[endpoints.transport] type = "postmark"` | `transport postmark { ... }` |
| `${env.POSTMARK_API_KEY}` | `{env.POSTMARK_API_KEY}` |

The internal config struct is the same. The Caddyfile parser produces an identical `core.Config` to what the TOML loader produces — parity tests assert this in CI.

## Multiple endpoints in one Caddyfile

```nginx
example.com {
    posthorn /api/contact {
        to alerts@example.com
        subject "Contact: {{.name}}"
        body templates/contact.txt
        transport postmark { api_key {env.POSTMARK_CONTACT_KEY} }
    }

    posthorn /api/newsletter {
        to list@example.com
        subject "Subscribe: {{.email}}"
        body "{{.email}} subscribed."
        transport postmark { api_key {env.POSTMARK_BROADCAST_KEY} }
    }

    file_server
}
```

## File path body templates

If the `body` value doesn't contain `{{`, it's treated as a file path relative to Caddy's working directory:

```nginx
body templates/contact.txt
```

```text
# templates/contact.txt
From: {{.name}} <{{.email}}>

{{.message}}
```

Inline strings work too (anything with a `{{` is parsed as a template directly):

```nginx
body "Hi {{.name}}, thanks for reaching out."
```

## Validating

```bash
./caddy validate --config Caddyfile
```

The adapter implements `caddy.Validator`, so `caddy validate` will catch missing required fields, invalid template syntax, and unresolved `{env.VAR}` placeholders before reload.

## Why have an adapter at all?

The brief commits to a Caddy adapter because Caddy users represent a meaningful slice of the audience (the project's prior life was a Caddy-only `mailout` replacement). Native Caddyfile syntax beats running a separate container for them.

But the architecture explicitly keeps Caddy **out of the core module** — the standalone gateway has zero Caddy dependency. The adapter is a thin (~150 LOC) wrapper that lives in a separate Go module and depends on both Caddy and the core.

See [ADR-6 and ADR-7](https://github.com/craigmccaskill/posthorn/blob/main/spec/03-architecture.md#architectural-decisions-log) for the architectural reasoning.
