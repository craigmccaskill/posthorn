This walks through standing up a working contact form on Docker against a real transactional-mail account. It assumes you've already followed [Installation](/getting-started/installation/) and have a provider API key in hand.

<Aside type="note" title="Transport choice">
This walkthrough uses **Postmark** as the canonical example. **Resend, Mailgun, AWS SES, and outbound-SMTP relay** are also supported — the walkthrough structure doesn't change, just the `[endpoints.transport]` block at step 2. See [Transports](/configuration/transports/) for the per-provider config.
</Aside>

<Steps>

1. **Create a working directory.**

   ```bash
   mkdir posthorn-test && cd posthorn-test
   ```

   <FileTree>
   - posthorn-test/
     - docker-compose.yml
     - posthorn.toml
     - .env
   </FileTree>

2. **Write `posthorn.toml`.**

   ```toml
   [[endpoints]]
   path = "/api/contact"
   to = ["you@example.com"]
   from = "Contact Form <noreply@example.com>"
   honeypot = "_gotcha"
   required = ["name", "email", "message"]
   subject = "Contact from {{.name}}"
   body = """
   From: {{.name}} <{{.email}}>

   {{.message}}
   """

   [endpoints.transport]
   type = "postmark"

   [endpoints.transport.settings]
   api_key = "${env.POSTMARK_API_KEY}"

   [endpoints.rate_limit]
   count = 5
   interval = "1m"
   ```

   Replace `you@example.com` with your inbox and `noreply@example.com` with an address on your Postmark-verified sending domain.

3. **Write `.env`.**

   ```bash
   POSTMARK_API_KEY=your-postmark-server-token-here
   ```

   <Aside type="caution" title="Treat this like a database password">
   For Postmark specifically: the *server token* is per-server, not per-account — find it under your server's API Tokens tab. Other providers have analogous scoped credentials.

   Standard hygiene applies: `.env` in `.gitignore`, file perms locked down, secret stays in env vars (never inlined in TOML), rotated on any suspicion of leak. See [API keys & secrets](/security/api-keys/) for the in-process guarantee and the rotation runbook.
   </Aside>

4. **Write `docker-compose.yml`.**

   ```yaml
   services:
     posthorn:
       image: ghcr.io/craigmccaskill/posthorn:latest
       restart: unless-stopped
       volumes:
         - ./posthorn.toml:/etc/posthorn/config.toml:ro
       env_file: .env
       ports:
         - "8080:8080"
   ```

5. **Start the container.**

   ```bash
   docker compose up -d
   docker compose logs -f posthorn
   ```

   You should see JSON log lines like:

   ```json
   {"time":"2026-05-16T20:00:00Z","level":"INFO","msg":"posthorn starting","version":"v1.0.0","listen":":8080","config":"/etc/posthorn/config.toml","endpoints":1}
   {"time":"2026-05-16T20:00:00Z","level":"INFO","msg":"endpoint registered","path":"/api/contact","transport":"postmark","recipients":1}
   {"time":"2026-05-16T20:00:00Z","level":"INFO","msg":"http ingress listening","addr":":8080"}
   ```

6. **Submit a test form.**

   ```bash
   curl -i -X POST http://localhost:8080/api/contact \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "name=Test User" \
     -d "email=test@example.com" \
     -d "message=hello from curl"
   ```

   You should receive `HTTP/1.1 200 OK` with a JSON body:

   ```json
   {"status":"ok","submission_id":"7f2c84d6-9b1e-4c2f-a3b8-1a2b3c4d5e6f"}
   ```

   Back in the terminal where you ran `docker compose logs -f posthorn` (step 5), a structured log line appears:

   ```json
   {
     "time": "2026-05-16T20:01:23Z",
     "level": "INFO",
     "msg": "submission_sent",
     "submission_id": "7f2c84d6-9b1e-4c2f-a3b8-1a2b3c4d5e6f",
     "endpoint": "/api/contact",
     "transport": "postmark",
     "latency_ms": 312,
     "transport_message_id": "0a1b2c3d-postmark-id"
   }
   ```

   `transport_message_id` lets you pivot from Posthorn's log straight to the message in the Postmark dashboard. See [Reading the logs](/deployment/reading-logs/) for how to filter, ship, and recover failed submissions from the log stream.

7. **Check your inbox.** The email should arrive within seconds. Subject line will be `Contact from Test User`. Body will be the rendered template.

   <Aside type="caution" title="If the email lands in spam (or never arrives)">
   The first deliverability gotcha for almost every new Posthorn deployment is **DNS on the sending domain**. Postmark sends *from* `noreply@example.com` (whatever you configured), and recipient providers will quarantine that mail unless your domain has SPF, DKIM, and DMARC records correctly set up.

   This is a five-minute fix done at your DNS host, not in Posthorn config. **[Set up SPF, DKIM, and DMARC →](/security/dns/)**
   </Aside>

</Steps>

## Wiring it into a real form

The form action POSTs to `/api/contact` on Posthorn. For a static site behind Caddy:

```html
<form method="POST" action="/api/contact">
  <label>Name <input name="name" required></label>
  <label>Email <input name="email" type="email" required></label>
  <label>Message <textarea name="message" required></textarea></label>
  <input type="text" name="_gotcha" tabindex="-1" autocomplete="off"
         style="position:absolute;left:-9999px">
  <button type="submit">Send</button>
</form>
```

The `_gotcha` field is the honeypot — visible to bots, invisible to humans. Any non-empty value silently 200s the request without sending mail.

For a Caddy reverse proxy in front of Posthorn:

```nginx
example.com {
    reverse_proxy /api/contact* posthorn:8080
    file_server
}
```

- Add a custom [success/error redirect](/features/templating/#redirects) for non-JS forms
